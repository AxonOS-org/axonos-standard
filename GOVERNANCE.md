# Governance

AxonOS governance is designed around technical clarity, safety discipline, and evidence-tagged evolution.

## Governance objectives

- Preserve a stable BCI software standard.
- Prevent claim inflation.
- Maintain clear reference implementation boundaries.
- Keep privacy, consent, and real-time behavior reviewable.
- Separate experimental work from normative architecture.

## Change process

Changes to the standard should be introduced through RFCs.

A standard-impacting change should include:

1. motivation;
2. affected components;
3. safety and privacy impact;
4. validation impact;
5. migration notes;
6. implementation status.

## Repository status categories

| Status | Meaning |
|---|---|
| Normative | Defines part of the AxonOS standard |
| Reference | Implements part of the standard |
| Experimental | Explores a future standard extension |
| Integration | Bridges AxonOS with external hardware/software |
| Historical | Preserved for audit or compatibility context |

## Claims policy

No repository should claim:

- certification without certification;
- hardware validation without trace artifacts;
- clinical readiness without clinical evidence;
- affiliation without explicit agreement;
- full formal verification when only bounded checks exist.
