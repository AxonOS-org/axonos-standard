1. PURPOSE

This document defines the public technical due-diligence position of AxonOS.

It is written for technical investors, external engineering reviewers, formal-methods reviewers, safety-critical systems reviewers, neurotechnology infrastructure partners, and early regulatory advisors.

This document is not a marketing brochure.

This document is not a clinical validation report.

This document is not a regulatory submission.

This document is not an FDA clearance, CE mark, IEC 62304 certification, ISO 14971 risk-management file, ISO 13485 quality-system certificate, or medical-device approval claim.

The purpose of this document is to state, in precise technical language, what AxonOS currently is, what it currently claims, what it does not claim, what evidence is currently public, and what must still be completed before any safety-critical or clinical deployment claim can be made.


2. EXECUTIVE POSITION

AxonOS is a pre-clinical cognitive operating-system architecture for brain-computer interface systems.

The project defines and implements the missing software layer between neural hardware and AI-enabled applications.

The intended system boundary is:

    neural hardware
        -> acquisition boundary
        -> real-time kernel substrate
        -> consent and neural-permission enforcement
        -> deterministic intent processing
        -> typed neural intent events
        -> application SDK
        -> assistive, research, or intelligent applications

The central AxonOS thesis is that BCI systems require a dedicated operating layer, not merely application libraries, AI agents, GUI tools, or ad-hoc device drivers.

That operating layer must provide deterministic timing boundaries, explicit consent-state enforcement, capability-based access control, typed intent-event boundaries, raw neural data minimization, auditability, compatibility governance, evidence-level separation, and safety-critical review discipline.

The current public AxonOS project demonstrates serious architectural discipline and a credible technical substrate.

The current public AxonOS project is not yet a clinical-grade safety case.

The immediate priority is canonicalization and evidence governance:

    1. one standard authority;
    2. one compatibility matrix;
    3. one claims registry;
    4. one review policy;
    5. one security disclosure policy;
    6. explicit repository roles;
    7. explicit evidence levels;
    8. no claim beyond available evidence.


3. AXONOS DEFINITION

AxonOS is a cognitive operating-system architecture for brain-computer interface systems.

It is designed to sit between neural acquisition hardware, embedded real-time processing components, consent and neural-permission infrastructure, AI-assisted interpretation layers, and application-facing SDKs.

AxonOS is an operating-system-layer project.

AxonOS is not merely an AI-agent framework.

AxonOS is not a chatbot runtime.

AxonOS is not a wellness application.

AxonOS is not a speculative token project.

AxonOS is not a generic SDK.

AxonOS is not merely a fork of existing BCI GUI software.

AxonOS is not, by itself, a medical-device approval claim.

The project is best understood as a BCI-native operating substrate: a set of standards, protocols, reference implementations, and verification practices for transforming neural signals into permissioned, typed, auditable intent-level events.


4. CORE TECHNICAL THESIS

Classical computing required operating systems before large application ecosystems could exist safely and predictably.

BCI systems require an analogous operating layer because neural data is not ordinary application data.

Raw neural signals may encode, imply, or correlate with motor intent, attention, fatigue, error recognition, affective state, cognitive workload, biometric patterns, and private mental or physiological state.

Therefore, application-level access to neural data must not be treated like access to an ordinary sensor stream.

AxonOS asserts the following operating principle:

    Applications should consume typed, consent-bound, temporally valid intent events, not unrestricted raw neural streams.

This principle drives the architecture:

    raw neural signal
        -> bounded acquisition epoch
        -> preprocessing and quality gate
        -> deterministic classification boundary
        -> typed intent event
        -> consent reference
        -> capability check
        -> application delivery

The purpose of AxonOS is to make this boundary explicit, inspectable, enforceable, and eventually certifiable.


5. REPOSITORY-LEVEL ARCHITECTURE

AxonOS is organized as a multi-repository system. Each repository has a distinct role and must not overclaim outside that role.

Repository roles:

    axonos-standard
        Canonical standard authority.
        Defines AOS artifacts, evidence posture, claims governance, and compatibility policy.

    axonos-rfcs
        Engineering RFC process.
        Records proposals, design decisions, and architectural changes.

    axonos-kernel
        Real-time kernel substrate.
        Contains scheduler, IPC, capability model, timing-oriented primitives, and kernel-level architecture.

    axonos-sdk
        Application boundary.
        Contains typed intent API, SDK boundary, and ABI-facing application structures.

    axonos-consent
        Consent reference implementation.
        Contains consent FSM, wire event model, and neural-permission enforcement core.

    axonos-swarm
        Distributed timing research.
        Experimental coordination, timing, and mesh-related research.

    axon-bci-gateway
        Acquisition boundary.
        OpenBCI GUI integration fork for hardware-in-the-loop and acquisition workflows.

    .github
        Organization profile and public governance surface.

No single implementation repository is the complete AxonOS system.

The safety case emerges only from the relationship between standard, RFCs, implementation, tests, validation, and review process.


6. EVIDENCE MODEL

AxonOS uses an evidence-level model to prevent overclaiming.

Evidence levels:

    L0
        Design intent or architecture statement.

    L1
        Static proof, model checking, type-level guarantee, or deterministic unit proof.

    L2
        Software test, fuzzing, simulation, property test, or CI result.

    L3
        Hardware-in-the-loop or measured physical validation.

    L4
        Independent third-party audit or laboratory validation.

The current public AxonOS stack primarily contains L0, L1, and L2 evidence.

The current public AxonOS stack does not yet provide a complete L3 or L4 safety case.

Every substantial claim must answer:

    1. What exactly is being claimed?
    2. Which repository owns the claim?
    3. Which file or standard defines it?
    4. Which evidence level supports it?
    5. What would falsify it?
    6. Is the result derived, simulated, model-checked, or physically measured?
    7. Is the evidence internal or independently reviewed?

8. TECHNICAL STRENGTHS

The current project has several strong technical properties.

First, AxonOS separates standard, RFCs, kernel, SDK, consent, gateway, and experimental swarm work. This prevents acquisition code, application code, consent semantics, and safety claims from collapsing into a single unreviewable repository.

Second, the project is Rust-first where appropriate and appears oriented toward no_std, small trusted computing base, explicit unsafe-surface control, deterministic data representation, and bounded execution paths.

Third, consent is treated as a system-level primitive, not merely an application preference. In BCI systems, consent withdrawal must be enforceable below the application layer.

Fourth, the typed intent-event model reduces raw-data exposure and creates a structured boundary between signal processing and applications.

Fifth, AxonOS uses evidence-level language. This is important because it separates design intent, simulation, model checking, hardware measurement, independent audit, and clinical validation.

Sixth, the gateway repository occupies a limited acquisition-boundary role and should remain explicitly non-safety-critical.


9. CURRENT TECHNICAL RISKS

The strongest current organizational risk is evidence governance drift.

This includes version drift, claim drift, standard-to-code drift, repository role ambiguity, and insufficient independent review trail.

The consent layer requires especially careful hardening because it sits at the neural-permission boundary. Any terminology around integrity, authentication, signature verification, secure-element attestation, persistence, and audit logging must be exact.

The kernel layer requires exact timing evidence discipline. Timing claims must state target hardware, compiler profile, optimization level, toolchain version, measurement method, interrupt assumptions, cache assumptions, scheduling assumptions, and whether the result is WCET, WCRT, average latency, or jitter.

The SDK layer requires strict invalid-data handling. Invalid wire data should be rejected, not silently normalized into ambiguous application-level semantics unless explicitly specified and tested.

The swarm layer must remain experimental unless distributed timing claims are formally modeled and verified.

The organization must reduce solo-maintainer risk through visible review process, CODEOWNERS, external reviewers, and traceable review records.

13. PRIORITIZED REMEDIATION PLAN

Phase 0: Canonical public surface.

    1. Keep axonos-standard as the canonical authority.
    2. Maintain compatibility matrix.
    3. Maintain claims registry.
    4. Maintain review policy.
    5. Remove repository-role ambiguity.
    6. Keep all AOS artifacts in one canonical location.
    7. Keep public non-claims visible.

Phase 1: Claim hardening.

    1. Review all README files for clinical, regulatory, cryptographic, WCET, WCRT, and formal-verification language.
    2. Map every strong claim to evidence level.
    3. Downgrade unsupported claims to design intent.
    4. Remove ambiguous signature wording where only integrity is implemented.
    5. Clarify experimental status of swarm.
    6. Clarify acquisition-only status of gateway.
    7. Clarify pre-clinical status of consent and kernel.

Phase 2: Consent hardening.

    1. Replace or clearly reclassify non-cryptographic integrity tag.
    2. Use constant-time primitives appropriate to the actual claim.
    3. Replace fail-stop invalid-state handling with fail-safe Withdrawn.
    4. Make wire fields private or validated.
    5. Add ABI layout annotations where appropriate.
    6. Define persistence trait.
    7. Define audit sink trait.
    8. Synchronize SDK consent protocol version.
    9. Add conformance vectors.

Phase 3: Kernel evidence discipline.

    1. Define timing-evidence profile.
    2. Separate derived, simulated, model-checked, and measured timing claims.
    3. State compiler profile assumptions.
    4. State target hardware assumptions.
    5. Add hardware measurement plan.
    6. Ensure Kani claims are not overstated as hardware timing proof.
    7. Add response-time-analysis notes where applicable.

Phase 4: Independent review.

    1. Add CODEOWNERS.
    2. Recruit external reviewer.
    3. Add Reviewed-by trailers.
    4. Add review classes.
    5. Preserve review trail.
    6. Add security disclosure workflow.
    7. Add SBOM generation.
    8. Add dependency review process.

Phase 5: L3 and L4 validation preparation.

    1. Define hardware-in-the-loop validation protocol.
    2. Define risk-management file skeleton.
    3. Map hazards to mitigations.
    4. Map mitigations to tests.
    5. Map tests to evidence levels.
    6. Prepare external lab validation plan.
    7. Prepare regulatory advisor review package.

15. FINAL POSITION

AxonOS is not yet a medical-device safety case.

AxonOS is not yet a clinical product.

AxonOS is not yet a regulatory package.

But AxonOS is a serious attempt to define the missing operating-system layer for the BCI era.

The project value lies in the correctness of its architectural question:

    Who controls the boundary between neural signal, machine inference, consent, and application action?

AxonOS answers:

    That boundary must be an operating system.

The next stage is to prove that answer with disciplined evidence.
